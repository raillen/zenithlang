# Decision 014 - Borealis Editor Preview IPC

- Status: accepted
- Date: 2026-04-22
- Type: architecture / editor / preview bridge / IPC
- Scope: Borealis editor to runtime-preview communication

## Summary

The first preview bridge for the Borealis editor will use:

1. a separate preview process;
2. `stdio` as the first transport;
3. newline-delimited JSON messages;
4. versioned envelopes;
5. request, event, response, and error channels.

## Decision

Adopt this first preview IPC model:

1. The editor launches the preview as a sidecar/native child process.
2. The editor writes JSON lines to the preview `stdin`.
3. The preview writes JSON lines to `stdout`.
4. Each message is one complete JSON object per line.
5. Messages use a versioned envelope.
6. The first protocol must be human-readable and easy to log.

## Why `stdio` first

This is the smallest reliable starting point.

It helps us:

- avoid premature socket/server complexity;
- keep launch flow simple;
- record and replay messages easily;
- debug the bridge with plain logs.

If later we need long-lived bidirectional streaming with more flexibility, the same message model can move to a local socket.

## Canonical Transport Name

The first transport is:

- `jsonl-stdio`

Meaning:

- JSON Lines
- over standard input/output

## Envelope Shape

Every message must follow this logical structure:

- `protocol`
- `seq`
- `channel`
- `kind`
- `payload`

## Canonical Rules

### `protocol`

- integer protocol version;
- first version is `1`.

### `seq`

- monotonic message sequence number generated by the sender.

### `channel`

Allowed values:

- `command`
- `event`
- `response`
- `error`
- `heartbeat`

### `kind`

- short stable message name;
- example: `hello`, `open_project`, `enter_play_mode`.

### `payload`

- structured object payload;
- must remain additive and version-friendly when possible.

## First Message Set

### Editor to preview

- `hello`
  - starts handshake and shares editor identity.
- `open_project`
  - asks preview to load a project.
- `open_scene`
  - asks preview to load a scene document.
- `enter_play_mode`
  - starts runtime execution.
- `stop_play_mode`
  - stops runtime execution.
- `select_entity`
  - sends current selected stable id.
- `focus_entity`
  - asks preview camera to focus selected entity.
- `ping`
  - liveness check.

### Preview to editor

- `hello`
  - handshake response from preview.
- `status`
  - current preview state such as `idle`, `loading`, `ready`, `playing`.
- `diagnostic`
  - error, warning, or informational report.
- `selection_changed`
  - preview-originated selection event.
- `frame_stats`
  - frame time, fps, and timing data.
- `pong`
  - liveness response.

## Example Message

```json
{"protocol":1,"seq":7,"channel":"command","kind":"open_scene","payload":{"path":"scenes/sample.scene.json"}}
```

## Handshake Flow

Recommended first flow:

1. editor launches preview process;
2. editor sends `hello`;
3. preview replies with `hello`;
4. editor sends `open_project`;
5. preview reports `status=loading`;
6. preview reports `status=ready`.

## Error Model

Errors must not be hidden.

The preview should emit `channel="error"` messages for:

- invalid command;
- project load failure;
- scene load failure;
- runtime crash boundary;
- unsupported feature;
- serialization/parse failure.

The editor should log and show these clearly.

## Logging Rule

All raw IPC lines should be easy to capture in logs during development.

This is one of the main reasons to keep the first transport text-based.

## Evolution Rule

The message schema should evolve like this:

1. prefer adding fields;
2. avoid breaking message names too early;
3. bump `protocol` only for real incompatible changes.

## Migration Path

If `stdio` becomes limiting, the next transport should be:

- local socket

The message envelope and message kinds should remain the same.

That means transport may change later without forcing a full editor-preview protocol rewrite.

## Out Of Scope

This decision does not yet finalize:

1. remote/networked preview;
2. collaborative editing transport;
3. binary transport;
4. authentication or security layers outside the local machine flow.
