## 2. Divergências Classificadas (P0/P1/P2)

### P0 — Quebram coerência de ponta a ponta

| ID | Descrição | Spec | Código afetado | Impacto |
| --- | --- | --- | --- | --- |
| **D-01** | `check(...)` não existe como intrínseco do core implícito. Funciona via `std.test` import, mas spec diz que é canonical assertion helper sem import. | `surface-syntax.md` L639 | `compiler/semantic/types/checker.c` — não registra `check` como intrinsic; `compiler/targets/c/emitter.c` — sem lowering direto | Testes com `attr test` + `check(...)` funcionam via `std.test`, mas usar `check(...)` fora do contexto de teste/sem import falharia. Ambiguidade sobre se é core ou stdlib. |
| **D-02** | `Comparable<Item>` não está no prelude implícito. Spec lista 4 core traits: `Equatable`, `Hashable`, `Comparable`, `TextRepresentable`. Binder registra apenas 3 (sem `Comparable`). | `surface-syntax.md` L541-544 | `compiler/semantic/binder/binder.c` L644: `core_traits[] = {"Equatable","Hashable","TextRepresentable",NULL}`; `checker.c` L549-553: `zt_type_is_core_trait_name()` idem | User code que use `where T is Comparable` compila mas não resolve corretamente. |
| **D-03** | Operadores relacionais `< <= > >=` usam regra hardcoded de tipo (numerico + text) em vez de trait `Comparable`. | `surface-syntax.md` L360, `decisions/027-core-prelude-traits-and-operator-semantics.md` | `checker.c` L2223-2235: hardcoded `zt_type_is_numeric()`, `ZT_TYPE_TEXT` | User-defined structs com `apply Comparable` não podem usar `<`/`>` etc. |
| **D-04** | `core.Error` pode ser referenciado como tipo (builtin no checker L410), mas sua construção por caminho qualificado (`core.Error(...)`) é um workaround no binder, sem suporte limpo no emitter. | `stdlib-model.md`, `decisions/050-core-stdlib-boundary.md` | `compiler/semantic/binder/binder.c` L91: listado como builtin; `emitter.c`: nenhuma referência a `core.Error` | Pode falhar em cenários de construção mais complexos fora do happy path. |
