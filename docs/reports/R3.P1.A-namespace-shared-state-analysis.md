# R3.P1.A - Namespace Shared State: Future Analysis

- Date: 2026-04-23
- Status: published (analysis only, no implementation change)
- Type: language / concurrency / visibility / evolution
- Upstream: `docs/planning/roadmap-v3.md`, `docs/planning/checklist-v3.md`
- Decision base: `language/decisions/086-namespace-public-var-and-controlled-mutation.md`
- Downstream: `R3.M2` (concurrency base)

## Purpose

Prepare the safe evolution of `public var` for scenarios with concurrency and larger scale, without introducing `global` semantics and without breaking the rule that `public` is visibility, not lifetime.

This is an analysis artifact, not an implementation milestone. It feeds the semantics proposal of `R3.M2`.

## 1) Risk Analysis Of Shared Namespace State

`public var` is convenient but introduces shared state concerns that must be tracked explicitly.

Primary risks (ordered by impact):

1. **Execution order coupling**
   - function output may depend on call order when namespace state is mutated by helpers
   - effect: callers silently observe different results for identical inputs
   - example surface: `std.random.next()` return depends on prior mutation of `last_seed`/`draw_count`
2. **Test isolation loss**
   - suites become order-sensitive when state is not reset between scenarios
   - effect: flaky or implicit-order tests, parallel runners may expose races even in single-thread today
3. **Hidden mutation via helpers**
   - mutation can be buried inside `public func` that callers perceive as pure
   - effect: reasoning about a callsite requires reading the whole namespace
4. **Debug surface growth**
   - reproducing a failure requires capturing the full namespace state snapshot
   - effect: debugging moves from local (`params/return`) to global (`alias.*`)
5. **Concurrency hazards (future)**
   - today Zenith does not expose tasks officially, but `R3.M2` will
   - effect: a `public var` mutated by two tasks without synchronization is a data race by construction
6. **FFI boundary amplification (future)**
   - external C code may observe partial writes to namespace storage
   - effect: undefined behavior if the C side assumes atomic reads

Risks explicitly **out of scope** for this analysis:

- reexport behavior of namespace `var` across package boundaries;
- capability-style access control (`who can read a mutable public var`);
- observability hooks (instrumentation of namespace writes) as a language feature.

## 2) Proposal: `public var` Under Concurrency

Base model for `R3.M2`:

- **Default model: single-owner namespace state**
  - a `public var` is owned by the namespace declaring it
  - the single-owner rule is the execution contract for `public var` until an explicit wrapper is used
  - no automatic cross-task sharing is granted by `public var` alone
- **Task boundary: copy by default**
  - on `task` spawn, a `public var` is visible only to the spawning task unless routed through an explicit primitive
  - mutation inside a child task does not propagate to the parent by default
  - this keeps `task` + `channel` semantics predictable (aligned with `R3.M2` entries)
- **Shared opt-in: explicit wrapper required**
  - cross-task shared mutation requires an explicit `Shared<T>` (or equivalent primitive) declared at the variable site
  - a bare `public var` cannot be promoted to `Shared<T>` implicitly
- **Determinism rule**
  - for identical input and identical external state, a function reading `public var` should still return identical output in a single-task run
  - this is required to keep `R3.M2` determinism tests tractable

Rationale:

- this preserves the R3 guarantee that `public` is visibility, not concurrent access
- it pushes the complexity into explicit opt-in types, not into the default keyword surface
- it reuses the `task` + `channel` contract already planned in `R3.M2`

## 3) Proposal: Synchronization Primitives (Future)

Direction only, not a commitment for this cut:

- **`atomic<T>`** for primitive scalar types (`int`, `bool`, pointers) under concurrent access
  - read/write are single indivisible operations
  - not a substitute for channel-based coordination
- **`Shared<T>`** as the canonical mutable-shared wrapper
  - explicit at declaration site
  - requires synchronization discipline documented in spec
- **`OnceCell<T>` / `LazyLocked<T>`** for initialization-once patterns
  - preserves deterministic module init while allowing lazy cost
- No raw mutex/unlock surface yet
  - wrappers are the primary API; raw locks remain an advanced concern for a later cut

Explicit non-goals:

- no `global` keyword, ever;
- no implicit promotion from `public var` to `Shared<T>`;
- no implicit `atomic` semantics for ordinary `public var`.

## 4) Migration Guideline For Stdlib And Packages

Rule of thumb for adopting `public var`:

Use `public var` only when all of the following are true:

1. the value represents legitimate module-level runtime state (counters, caches, feature toggles, observability);
2. mutation is **contained inside the owner namespace** through `public func` APIs;
3. a short reset/observability helper exists for tests (`reset()`, `stats()`);
4. documentation names the invariants and the allowed mutation points;
5. the module is **not pure by contract** (pure modules stay pure).

Avoid `public var` when:

- the value is configuration that should be explicit per call (e.g. `timeout`, `now()` policy);
- the value is pure data transform state (`std.text`, `std.format`, `std.math`, `std.fs.path`);
- the value would create a global policy knob that tests cannot isolate;
- the value would replace a proper instance-based API (for example, `random.Generator(seed)` is better than a shared RNG state for serious workflows).

Checklist for stdlib/package authors:

- [ ] is mutation truly module-scoped?
- [ ] is there a `public func` for every mutation point?
- [ ] is a reset/observability helper documented and tested?
- [ ] do ZDoc comments name invariants and side effects?
- [ ] is the read surface (`alias.var`) intentional and documented?
- [ ] is there at least one positive and one negative behavior test?

Current conformant example: `stdlib/std/random.zt` (`seeded`, `last_seed`, `draw_count`, `stats()`).

## 5) Criteria To Promote This Analysis To An Implementation Milestone

Promote to milestone only when all are true:

1. **`R3.M2` concurrency base executable** with `task` + `channel` semantics stable;
2. **At least one real consumer** (stdlib or package) demonstrates a concrete need beyond `std.random`;
3. **Behavior coverage ready** for race/order/cancel scenarios involving namespace state;
4. **Benchmarks** confirm the chosen wrapper (`Shared<T>` / `atomic<T>`) stays inside the R3 perf budget;
5. **Spec text** published for the chosen wrappers and for the `public var` concurrent semantics;
6. **No P0 open** on `R3.M2` at the time of promotion.

When all 6 are satisfied, this analysis is elevated into a dedicated implementation milestone (provisional id `R3.Mx.sharedstate`) inside the active cycle.
