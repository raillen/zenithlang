# R3.M4 - Dyn Dispatch Minimum: Phase 1 Report

- Date: 2026-04-23
- Status: Phase 1 complete (foundation), Phases 2-3 deferred
- Scope: dyn<Trait> spec, decision, runtime structures, type checker validation, emitter type resolution, vtable infrastructure (code ready, wrapper generation pending)

## Summary

R3.M4 Phase 1 establishes the foundational infrastructure for dynamic dispatch via `dyn<Trait>` in Zenith. This includes the authoritative spec, decision record, runtime vtable/fat-pointer structures, type checker subset validation, C emitter type resolution for `dyn<Trait>` and `list<dyn<Trait>>`, and vtable generation code (structurally complete, requires wrapper functions for full type compatibility).

Full vtable generation with wrapper functions, boxing code generation, and dynamic method dispatch are deferred to follow-up phases within the R3 cycle.

## Deliverables

### 1. Decision 088 - Dyn Dispatch Minimum Subset

`language/decisions/088-dyn-dispatch-minimum-subset.md`

Defines the official subset rules for `dyn<Trait>`:
- No mutating methods (`mut func` blocked)
- Max 8 methods per trait
- All params/returns must be copyable
- No generic traits with `dyn`
- Fat-pointer representation (data + vtable)
- Boxing takes ownership, RC-managed

### 2. Spec - Dynamic Dispatch

`language/spec/dyn-dispatch.md`

Authoritative specification covering:
- Syntax: `dyn<Trait>`
- Fat pointer representation
- Vtable structure
- Boxing rules
- Method dispatch semantics
- Minimum subset (R3.M4)
- Ownership rules
- Performance characteristics

### 3. Runtime Structures

`runtime/c/zenith_rt.h` and `runtime/c/zenith_rt.c`

New types:
- `zt_vtable` - vtable with drop, clone, and method function pointers
- `zt_dyn_value` - fat pointer (data + vtable)
- `zt_list_dyn` - generic dyn list for heterogeneous collections

New functions:
- `zt_dyn_box()` - box a value with vtable
- `zt_dyn_unbox()` - extract data from dyn value
- `zt_dyn_get_vtable()` - get vtable from dyn value
- `zt_dyn_drop()` - RC-aware drop
- `zt_dyn_clone()` - deep clone via vtable
- `zt_list_dyn_create/append/get/free()` - list helpers

### 4. Type Checker Validation

`compiler/semantic/types/checker.c`

New function `zt_checker_validate_dyn_trait()`:
- Validates trait against subset rules
- Emits targeted diagnostics for violations:
  - `dyn.mut_method` - mutating method in dyn trait
  - `dyn.generic_trait` - generic trait with dyn
  - `dyn.too_many_methods` - >8 methods
  - `dyn.uncopyable` - non-copyable param/return

### 5. Diagnostics

`compiler/semantic/diagnostics/diagnostics.h` and `.c`

New diagnostic codes:
- `ZT_DIAG_DYN_MUT_METHOD`
- `ZT_DIAG_DYN_GENERIC_TRAIT`
- `ZT_DIAG_DYN_TOO_MANY_METHODS`
- `ZT_DIAG_DYN_UNCOPYABLE`
- `ZT_DIAG_DYN_NO_APPLY`
- `ZT_DIAG_DYN_FFI_UNSAFE`

All with ACTION/WHY/NEXT format for action-first diagnostics.

### 6. Emitter Type Resolution

`compiler/targets/c/emitter.c`

- Removed `dyn<TextRepresentable>` only restriction
- `dyn<Trait>` resolves to `zt_dyn_value *` for user traits
- `dyn<TextRepresentable>` retains legacy `zt_dyn_text_repr *` path
- `list<dyn<Trait>>` resolves to `zt_list_dyn *`
- Empty `list<dyn<Trait>>` creates via `zt_list_dyn_create()`

### 7. Vtable Generation Infrastructure

`compiler/targets/c/emitter.c`

- Vtable type definitions per trait (structurally complete)
- Vtable instances per (concrete_type, trait) pair (structurally complete)
- **Pending**: Wrapper function generation for type compatibility (method signatures differ from vtable void* signatures)
