## 1) Objective

Finalize action-first diagnostics as the default UX across all central paths, consolidate cognitive profiles, and cover the CLI flags `--focus`, `--since`, `summary` and `resume`.

## 2) Delivered

### 2.1 ACTION / WHY / NEXT Pattern

Gap closed: the action-first renderer previously emitted only `ACTION / WHY / SUGGESTION`. The normative pattern from `tools/decisions/005-tools-diagnostics-model.md` requires the explicit `NEXT` block.

Changes:

- `compiler/semantic/diagnostics/diagnostics.h`: declares `zt_diag_next_text(zt_diag_code)` as public API.
- `compiler/semantic/diagnostics/diagnostics.c`:
  - `zt_diag_next_text` returns a stable, code-specific next-step guidance for 25 diagnostic codes covering the central paths (name, type, mutability, control flow, project, docs).
  - `zt_diag_render_action_first` now emits the `NEXT:` block (with action arrow glyph on UTF-8 terminals) after the optional `SUGGESTION:` block and before the severity/effort summary line.
  - `zt_diag_render_ci` now emits `effort=<label>`, `action="..."`, `suggestion="..."`, and `next="..."` as stable additional fields on the single-line CI output, preserving backward-compatible prefix (`severity[code] source:line:col ... message`).

### 2.2 Profiles Consolidated

Existing and validated:

- `beginner` caps at 3 errors per batch (`zt_cog_profile_error_limit`).
- `balanced` caps at 5 errors per batch.
- `full` is uncapped.
- `zt_cog_profile_from_text` accepts `beginner`, `balanced`, `full` and falls back to `balanced` for any unknown/null input.
- `--profile <level>` lock on `zt_driver_context_lock_profile` switches action-first on for `beginner`/`balanced` and keeps detailed rendering for `full`.

### 2.3 Effort Hint Calibration

Snapshot validated via unit tests:

| Effort | Sample codes |
|---|---|
| `quick fix` | `syntax.error`, `syntax.unexpected_token`, `name.unresolved`, `name.confusing`, `mutability.const_update`, `semantic.param_ordering`, `lexer.token_too_long`, `lexer.unterminated_string` |
| `moderate` | `type.mismatch`, `type.invalid_call`, `type.invalid_argument`, `project.unresolved_import`, `project.missing_entry`, `doc.unresolved_target` |
| `requires thinking` | `control_flow.non_exhaustive_match`, `project.import_cycle`, `type.integer_overflow`, `zir.parse`, `zir.verify`, `backend.c.emit`, `backend.c.legalize` |

### 2.4 CLI Flags Coverage

- `--focus <fragment>`: implemented (`zt_diag_matches_focus`) filters diagnostics by substring match of the source file path. Verified empirically: `--focus main.zt` keeps diagnostics; `--focus no_match_path` removes all output.
- `--since <git-ref>`: implemented (`zt_diag_matches_since` + `ctx->since_filter`); filters diagnostics whose source path matches any fragment of the since-delta list. Requires a valid git ref with changed files.
- `summary`: help topic + command branch wired via `zt help summary` and in the main dispatcher.
- `resume`: help topic + command branch wired via `zt help resume` and in the main dispatcher.
