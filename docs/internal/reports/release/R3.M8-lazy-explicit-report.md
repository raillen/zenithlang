# R3.M8 - Lazy Explicito

- Status: completo para o primeiro corte executavel
- Data: 2026-04-24
- Escopo: `lazy<T>` explicito no compiler, runtime `lazy<int>` one-shot, stdlib e testes

## Resultado

R3.M8 adiciona lazy sem mudar a regra mental padrao da linguagem.

Expressões comuns continuam eager.

O programador precisa escrever:

```zt
const later: lazy<int> = lazy.once_int(func() => compute_value())
const value: int = lazy.force_int(later)
```

## Entregue

- tipo `ZT_TYPE_LAZY` no sistema de tipos
- parsing/resolução de `lazy<T>` como tipo generico builtin
- lowering HIR de `lazy<T>`
- mapeamento C de `lazy<int>` para `zt_lazy_i64 *`
- runtime `zt_lazy_i64`
- modulo `std.lazy`
- teste positivo de ordem de avaliacao
- teste negativo de reuso invalido
- spec `language/spec/lazy.md`

## Semantica Atual

`lazy<int>` é one-shot.

Regras:

- criar lazy não executa o thunk
- `force_int` executa o thunk
- segundo `force_int` no mesmo valor falha com `runtime.contract`
- não existe lazy implicito
- não existe conversão automática de `T` para `lazy<T>`

## Limites Conhecidos

- runtime executavel cobre `lazy<int>` neste corte
- `lazy<T>` generico fica para evolucao futura
- iteradores lazy ainda não sao API publica executavel
- lazy reutilizavel/cacheado ainda não existe

## Evidencias

- `python build.py`: verde
- `zt run tests/behavior/lazy_explicit_order_basic/zenith.ztproj`: exit code 0
- `zt run tests/behavior/lazy_reuse_error/zenith.ztproj`: falha esperada com `error[runtime.contract] lazy value already consumed`

## Risco Residual

Baixo para o corte entregue.

O principal risco restante é expectativa de API generica completa. Isso está documentado como fora do primeiro corte executavel.
