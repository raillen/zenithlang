# R3.M3 - FFI 1.0 Report (Phase 1)

- Date: 2026-04-23
- Scope: M35.2 FFI 1.0
- Status: delivered (Phase 1 complete) - Phases 2-4 explicitly deferred
- Upstream: `docs/internal/planning/roadmap-v3.md`, `docs/internal/planning/checklist-v3.md`, `language/decisions/011-extern-c-and-extern-host.md`
- Downstream: `docs/internal/reports/R3-risk-matrix.md`

## 1) Framing

The `R3.M3` roadmap item enumerated ABI contract, ownership rules at the FFI boundary, automatic shielding, type matrix, negative signature tests, and E2E fixtures. The emitter already ships `c_emit_ffi_shielded_zir_call_statement` (retain/release around managed FFI args) and `c_type_to_c` type mapping. `R3.M3` consolidates that surface into an authoritative spec, extends E2E coverage, and formalizes the allowed/blocked type matrix.

Deferred beyond this cut per the spec's phase plan:

- arity negative fixtures at `extern c` call site;
- callbacks and function-pointer interop (requires `R3.M5`);
- ABI annotations / symbol renaming.

These are tracked in `docs/internal/reports/R3-risk-matrix.md` as `R3-RISK-020`, `R3-RISK-021`, `R3-RISK-022`.

## 2) Delivered

### 2.1 Authoritative Spec

`language/spec/ffi.md` (new) consolidates:

- syntax (namespace-scope `extern c` block with function decls only; no bodies, no renaming, no ABI annotations);
- ABI contract (host C toolchain ABI; no silent conversions; mismatched argument types emit `type.mismatch` at call site; unmappable types emit `backend.c.emit` / `C_EMIT_UNSUPPORTED_TYPE`);
- allowed / blocked type matrix (primitives + `text`/`bytes`/`list<T>`/`map<K,V>`/`optional<T>`/`result<T,E>`/`outcome<T,E>`; structs/enums-with-payload/callbacks/raw pointers blocked);
- ownership / lifetime rules at boundary (automatic `zt_retain`/`zt_release` shielding for managed args; managed returns handed to caller's ARC path; C side must not free or cache beyond call);
- diagnostics list (`type.mismatch`, `type.invalid_call`, `type.invalid_argument`, `backend.c.emit`, `project.unresolved_import`);
- FFI phase table and cross-references.

### 2.2 E2E Fixtures

Positive, new this milestone:

- `tests/behavior/extern_c_text_len_e2e`
  - declares `extern c func zt_text_len(value: text) -> int`;
  - passes a managed `text` to the extern and verifies the returned `int`;
  - emitter auto-inserts the shielding block (`zt_ffi_arg0 = greeting; if (... != NULL) { zt_retain(...); } length = zt_text_len(zt_ffi_arg0); if (... != NULL) { zt_release(...); }`) verified by inspecting the generated C;
  - runs to `exit 0`.

Negative, new this milestone:

- `tests/behavior/extern_c_struct_arg_error`
  - declares the same `extern c` block and passes a user-defined `Snapshot` struct where `text` is expected;
  - compiler rejects with `error[type.mismatch]` at the call site with `expected text but got Snapshot`;
  - ACTION / WHY / NEXT pattern inherited from `R3.M1` present in diagnostic output;
  - diagnostic contract pinned by `tests/fixtures/diagnostics/extern_c_struct_arg_error.contains.txt`.

Positive, pre-existing (inventory, unchanged):

- `tests/behavior/extern_c_puts_e2e` (`puts(text) -> int` roundtrip).

### 2.3 Suite Registration

Both new fixtures are registered in `tests/suites/suite_definitions.py`:

- `extern_c_struct_arg_error` added to `BEHAVIOR_CHECK_FAIL` set and to `BEHAVIOR_DIAGNOSTIC_FRAGMENT_FILES` map;
- `extern_c_text_len_e2e` added to `BEHAVIOR_RUN_PASS` set (and `std_concurrent_boundary_copy_determinism` from `R3.M2` registered in the same pass for hygiene).

### 2.4 Shielding Evidence

Empirical evidence from the generated C for `extern_c_text_len_e2e`:

```
{
    zt_text *zt_ffi_arg0 = greeting;
    if (zt_ffi_arg0 != NULL) { zt_retain(zt_ffi_arg0); }
    length = zt_text_len(zt_ffi_arg0);
    if (zt_ffi_arg0 != NULL) { zt_release(zt_ffi_arg0); }
}
```

This confirms:

- managed pointer kept alive across the call regardless of what the callee does with it;
- original caller binding unaffected;
- release on return path.

## 5) Acceptance Statement

Against the `R3.M3` acceptance criterion ("FFI 1.0 estavel e auditavel com cobertura positiva e negativa"):

- **Met for Phase 1**:
  - ABI contract documented with delimited scope (host toolchain ABI; no silent conversions);
  - ownership rules at the boundary specified and empirically verified (shielding block in generated C);
  - allowed/blocked type matrix published;
  - positive coverage in `extern_c_text_len_e2e` (managed arg + primitive return) and `extern_c_puts_e2e` (primitive return);
  - negative coverage in `extern_c_struct_arg_error` with pinned diagnostic fragment.
- **Deferred**: arity negatives, callback interop, ABI annotations.

`R3.M3` closes as **Phase 1 complete, Phases 2-4 tracked**, consistent with `language/spec/ffi.md`.

## 6) Residual Risk

- ABI stability across toolchains (MSVC / MinGW / gcc-linux) validated only on `gcc` Windows reference today; `R3-RISK-020` remains monitoring.
- The type matrix is enforced by the type system at the call site, but the declaration itself is not restricted to allowed shapes (a user could declare `extern c func foo(s: Snapshot)` and the error only surfaces at the call site or at emit time). Follow-up: push the check earlier during semantic validation of the `extern c` block.
- `R3-RISK-021` (long-lived C pointers violating shielding semantics) remains a runtime-only class of bug with no static enforcement.
