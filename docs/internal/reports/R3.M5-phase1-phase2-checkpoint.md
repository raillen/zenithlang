## Escopo do R3.M5 (Decision 089)

- tipo `func(T1,...) -> R` como tipo de primeira classe
- referência a função top-level como valor (sem captura; closures = R3.M6)
- compatibilidade estrutural de assinatura
- chamada indireta via callable
- integração FFI (`extern c`) para function pointer
- escape/ownership rules (sem vazar para `public var`, sem captura)
- testes positivos + negativos + FFI
- docs: Decision 089 + `language/spec/callables.md`

## Limitacoes honestas do estado atual

- **ZIR/emitter** ainda nao conhecem callable. Programas com callable
  passam no checker e no HIR mas falham no ZIR verifier (tipo
  `ZT_TYPE_CALLABLE` nao mapeado). Phase 3 necessaria.
- **Indirect call** cobre somente `op(args)` onde `op` eh um ident
  ligado a callable. Cadeias (`get_callable()(args)`) precisam de novos
  nos em HIR.
- **FFI boundary check** (Decision 089 "Permitted/Rejected" para
  extern c) ainda nao eh emitido no checker. O codigo estavel
  `callable.extern_c_signature` esta reservado mas nao ha sitio de
  emissao ainda.
- **Fixtures positivas** (`callable_basic`, HOF) nao adicionadas -
  requerem Phase 3 (codegen) para passar `zt check` inteiro
  (ZIR verify rejeita hoje).
- **Fixtures negativas**: 5 entregues em Phase 5a. Cobrem 5 dos 6
  diagnosticos novos; falta fixture para `callable.extern_c_signature`
  (depende de Phase 4).
- **Interaction com `apply`/metodos**: callable nao pode referenciar
  metodo associado (`self`-bound); mencionado no spec como
  non-goal para v1.

## Relacao com R3.M2

O `R3.M2 Phase 3` (`jobs.spawn(func, arg)` / `jobs.join(job)`) esta
bloqueado pela ausencia de function-as-value no emitter. Phase 2 do
R3.M5 da semantica correta para o checker mas nao libera codegen -
so Phase 3 completa libera `R3.M2 Phase 3`.

Rastreamento de risco:
- `R3-RISK-010` (task/channel oficial)
- `R3-RISK-011` (Shared<T> avancado)
- `R3-RISK-012` (testes de corrida/cancelamento)

Ver `docs/internal/reports/R3-risk-matrix.md`.
