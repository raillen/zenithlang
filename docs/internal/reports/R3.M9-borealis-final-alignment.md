# R3.M9 Borealis Final Alignment

Status: checkpoint published
Date: 2026-04-24

## Summary

Borealis remains aligned with the R3 language direction.

The final R3 release is no longer blocked by Borealis.

The fallback behavior test now uses the deterministic stub backend.

## Important Point

This is not a language-design conflict.

It is a release-readiness issue.

## Final Gate Signal

Previously failing suite item:

- `behavior/borealis_backend_fallback_stub`

Previously observed assertion:

- `key_down should start false`

## Interpretation

The failure was caused by the fallback test requesting the Raylib backend while a local Raylib runtime was available.

The test now requests backend `0`, the deterministic stub backend.

This matters because Borealis is the main package validating the language in a game/runtime context.

## Alignment With R3 Features

Borealis can benefit from R3 features, especially:

- `public var` for narrow namespace-level observability;
- callables and closures for callbacks;
- lambdas/HOFs for small data transforms;
- explicit lazy for delayed setup paths;
- FFI shielding for backend integration.

## Recommendation

After final R3 publication:

1. Keep Borealis as a release-alignment gate for future compiler/runtime changes.
2. Keep real Raylib coverage separate from deterministic fallback coverage.
3. Avoid environment-sensitive backend selection in fallback tests.
