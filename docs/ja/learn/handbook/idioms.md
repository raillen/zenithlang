# イディオムとベストプラクティス：Zenith標準

このドキュメントは、産業レベルのZenithコードの書き方を定義します。これらのパターンに従うことで、コードが他の開発者にとっても読みやすくなり、Ascensionコンパイラによって最適化されることが保証されます。

## 1. 命名と意図

Zenithは、データと構造を区別するために視覚的なパターンを使用します。

| 要素 | パターン | 例 |
| :--- | :--- | :--- |
| 変数と関数 | `snake_case` | `current_speed` |
| 定数 | `SCREAMING_SNAKE` | `MAX_GRAVITY` |
| 型と構造体 | `PascalCase` | `PlayerProfile` |

---

## 2. 型付け：推測よりも明快さ

*   **コンテキスト**: コンパイラは型を推論できますが、人間の脳にはアンカー（手がかり）が必要です。
*   **ルール**: モジュールレベルの変数や公開関数では、型を明示的に宣言してください。
*   **対比**: 型が予測できないJavaScriptやLuaとは異なり、Zenithでは型は契約（コントラクト）です。

```zt
-- 推奨
var level: int = 1

-- 回避（公開コードにおいて）
var level = 1
```

---

## 3. 安全なアンラップ (Safety First)

これは、Zenithの産業的な安定性において最も重要なポイントです。

*   **コンテキスト**: ヌル値や失敗の処理。
*   **ルール**: 常に `or` 演算子（デフォルト値）または `match` を優先してください。 `!` 演算子は、 `is_present()` によるチェックの後にのみ使用してください。
*   **対比**: Zenithは、すべてに `!` を使用する「クラッシュ指向プログラミング」を推奨せず、回復力（レジリエンス）を促進します。

```zt
-- ✅ 安全: 産業レベルのコード
var user = os.get_env("USER") or "admin"

-- ❌ 危険: システムパニックを引き起こす可能性があります
var user = os.get_env("USER")!
```

---

## 4. 継承よりも合成

*   **ルール**: クラスの継承階層を作ろうとしないでください。データ (`struct`) を作成し、振る舞い (`trait`) をアタッチしてください。
*   **利点**: 「菱形継承問題」を回避し、コードを100％個別にテスト可能にします。

---

## 5. ネイティブの制限 (Boundary Principle)

*   **コンテキスト**: ZenithはネイティブなLuaの使用を許可していますが、これは言語の主権を損なうことになります。
*   **ルール**: `native lua` は、以下の目的のためにのみ使用してください。
    1.  FFIを介した外部Cライブラリへのアクセス。
    2.  コアにまだ存在しない極端なループの最適化。
    3.  レガシーコードとの相互運用。
*   **対比**: 純粋なZenithコードは静的に検証可能ですが、ネイティブなLuaコードは「ブラックボックス」です。

## 6. 品質チェックリスト

モジュールを完成させる際、以下を確認してください。
- [ ] 失敗する可能性のある操作に `Outcome` を使用していますか？
- [ ] `where` 句は、変数を無効な値から保護していますか？
- [ ] 関数名は動詞的ですか (`data` ではなく `load_data`)？
- [ ] `.zt` ファイルをクリーンに保つために、ドキュメントは `.zdoc` ファイルに移動されましたか？
