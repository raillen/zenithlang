# R3.M2 Residual Scope Report

Status: residual scope open
Date: 2026-04-24
Owner: R3.M2 follow-up

## Summary

R3.M2 delivered the Phase 1 concurrency base.

R3.M2 did not deliver the full original milestone scope.

This matters for R3.M9 because the final release should not claim that the whole R3 concurrency milestone is complete unless the remaining scope is either implemented or formally moved out of R3.

## Delivered In R3.M2

- authoritative concurrency spec;
- boundary-copy contract;
- `std.concurrent.copy_*` helpers;
- deterministic copy behavior;
- namespace-state rules for `public var` under the current single-owner model.

## Still Open

| Item | Current Status | Risk |
|---|---|---|
| official `task` + `channel` model | not implemented | `R3-RISK-010` |
| runtime tests for race/order/cancellation | partial; order via copy covered, race/cancellation not covered | `R3-RISK-012` |
| explicit `Shared<T>` path | not implemented | `R3-RISK-011` |
| checker-level `transferable` predicate | partial groundwork only | tracked by R3.M2 follow-up |

## Release Impact

There are two valid paths.

Path A: finish R3.M2 before final R3 publication.

Path B: formally move the remaining R3.M2 scope to a later cycle and document that R3 shipped only the Phase 1 concurrency base.

Without one of these choices, the final R3 release would overstate its concurrency status.

## Recommendation

Use Path B unless we want to pause R3.M9 and implement a much larger concurrency surface now.

Reason:

- `task`;
- `channel`;
- cancellation;
- `Shared<T>`;
- and race tests

are not small polish items. They change runtime semantics and need their own design/review loop.

## Required Before Final R3 Publication

- choose Path A or Path B explicitly;
- update roadmap/checklist wording;
- keep R3.M9 final acceptance unchecked until this is resolved;
- mention the residual scope in release notes.
