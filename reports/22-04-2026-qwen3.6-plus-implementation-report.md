# Zenith Language — Deep Implementation Analysis Report

**Model:** qwen3.6-plus  
**Date:** 22-04-2026  
**Target Version:** 0.3.0-alpha.1 (R2.M12)  
**Scope:** Full compiler pipeline (lexer → parser → AST → binder → type checker → HIR → ZIR → verifier → C emitter → GCC/Clang → native binary), runtime C, stdlib  
**Constraint:** Read-only analysis — no code modifications

---

## 1. Architecture Overview

### 1.1 Pipeline

```
.zt source
  → lexer (tokenizer)
  → parser (AST construction)
  → semantic analysis (binder + type checker)
  → HIR (High-level IR, normalized)
  → ZIR (Zenith IR, backend-facing)
  → ZIR verifier
  → C emitter (code generation)
  → GCC/Clang (native compilation)
  → native executable
```

## 2. Language Surface Model

### 2.1 Type System

**Scalar types:** `bool`, `int` (fixed int64), `int8/16/32/64`, `u8/16/32/64` (uint aliases), `float` (fixed float64), `float32/64`, `text` (immutable UTF-8), `bytes` (immutable binary), `void`

**Generic types:** `list<T>`, `map<K,V>`, `optional<T>`, `result<T,E>`, `dyn<Trait>`

**Core implicit traits:** `Equatable`, `Hashable`, `TextRepresentable`, `Comparable`

### 2.2 Memory Model

- **Default:** Value semantics (copy on assignment/parameter passing)
- **Implementation:** Non-atomic ARC + COW for managed values
- **Managed values:** `text`, `bytes`, `list<T>`, `map<K,V>`, structs with managed fields, enums with managed payloads
- **No GC in MVP** — cycle detection is documented future work
- **`const` collections** are observably immutable (enforced at compile time)

### 2.3 Mutability Model

- `const` — immutable binding
- `var` — mutable local
- `mut func` — mutating methods (require `var` receiver)
- `public var` — namespace-level mutable state (read cross-namespace, write only in owner namespace)

### 2.4 Error Handling

- `panic(message)` — fatal, not catchable
- `optional<T>` + `none` — absence
- `result<T,E>` + `success`/`error` — recoverable failure
- `?` propagation operator — early return in functions returning `optional<...>` or `result<...>`
- `where` contracts — runtime panic on failure (does NOT change constructor return type)

### 2.5 Generics Strategy

- **Monomorphization** — creates specialized C code per type substitution
- Deduplication via string-set canonical type keys
- Configurable limit via `build.monomorphization_limit`
- `dyn Trait` — fat pointers for heterogeneous collections (vtable + data)

## 3. Compiler Component Analysis

### 3.1 Lexer (`compiler/frontend/lexer/lexer.c` — 489 LOC)

**Responsibilities:** Tokenize `.zt` source into typed tokens with spans

**Critical functions:**
| Line | Function | Risk |
|------|----------|------|
| 168 | `zt_lexer_read_string()` | No size limit on string literal; unbounded memory consumption |
| 221 | `zt_lexer_read_triple_quoted()` | Same unbounded read for triple-quoted text |
| 246 | `zt_lexer_read_number()` | Handles hex (`0x`), binary (`0b`), float with exponent |
| 340 | `zt_lexer_next_token()` | Main dispatch; handles `--- ... ---` block comments |

**Bugs found:**
- **LEX-01:** `zt_lexer_read_string()` reads until `"` or EOF with no size cap. A 10MB unclosed string literal consumes proportional memory.
- **LEX-02 (Parser-side):** `zt_parser_normalize_hex_bytes()` at `parser.c:284` uses `char normalized[1024]` stack buffer. Hex literals > 512 bytes truncate silently.
- **No escape validation:** `\x` and `\u` sequences are not validated for correctness — `\xGG` is accepted as `\x` + `GG`.

### 3.2 Parser (`compiler/frontend/parser/parser.c` — 2,034 LOC)

**Responsibilities:** Build AST from token stream with error recovery

**Critical functions:**
| Line | Function | Risk |
|------|----------|------|
| 348 | `zt_parser_find_fmt_expr_end()` | Brace/quote tracking for fmt interpolation — complex state machine |
| 284 | `zt_parser_normalize_hex_bytes()` | Fixed 1024-byte stack buffer |
| 212 | `ZT_PARSER_MAX_NAME_PATH_LEN` | 1024 char limit for namespace paths |
| 1190 | `zt_parser_parse_func_decl()` | Function parsing with params, defaults, generics |
| 1411 | `zt_parser_parse_enum_decl()` | Enum with payload variants |

**Bugs found:**
- **PAR-01:** `zt_parser_find_fmt_expr_end()` state machine for `in_string`/`in_triple`/`escaped` can fail with deeply nested `{}` inside strings within fmt interpolation. Example: `fmt "a { fmt \"b { 1 + 2 }\" } c"`
- **PAR-02:** `zt_parser_append_to_name_path()` uses `buf[ZT_PARSER_MAX_NAME_PATH_LEN + 1]` (1025 bytes). Namespace paths > 1024 chars truncate silently.
- **No recursion depth limit:** Deep nesting (2000+ levels) of `if`/`while`/`for` blocks can cause C stack overflow during recursive descent parsing.

### 3.3 AST (`compiler/frontend/ast/model.c` — ~540 LOC)

**Responsibilities:** AST node model, arena allocation

**Design notes:**
- Arena-based allocation — no per-node `free()`, entire AST freed at once
- Nodes are lightweight structs with unions for variant data
- String pool for identifier interning

### 3.4 Binder (`compiler/semantic/binder/binder.c` — ~692 LOC)

**Responsibilities:** Symbol table construction, import resolution, namespace scoping

**Key behavior:**
- Imports are namespace-level, not individual symbols
- `import std.io as io` — qualified access via `io.print()`
- Shadowing is rejected in MVP
- Duplicate names in same scope are errors

### 3.5 Type Checker (`compiler/semantic/types/checker.c` — 3,201 LOC)

**Responsibilities:** Type inference, validation, trait checking, assignability

**Critical functions:**
| Line | Function | Risk |
|------|----------|------|
| 533 | `zt_checker_compute_integral_binary()` | Signed overflow UB in constant folding |
| 836 | `zt_checker_same_or_contextually_assignable()` | Core assignability logic — 100+ lines |
| 1170 | `zt_checker_check_call_expr()` | Call resolution with named args, defaults, trait methods |
| 1829 | `zt_checker_check_expression()` | Main expression checker — massive switch |
| 237 | `zt_catalog_find_decl()` | O(n) linear search through declarations |

**Bugs found:**
- **SEM-03:** `zt_checker_compute_integral_binary()` computes `left + right` in `long long`. `9223372036854775807 + 1` is signed integer overflow — undefined behavior in C. No overflow detection before arithmetic.
- **SEM-04:** `zt_parse_double_literal_value()` uses `strtod()` — `1e309` produces `inf` without diagnostic.
- **SEM-05:** `zt_catalog_find_decl()` is O(n) linear scan. Projects with 2000+ declarations show quadratic degradation.
- **SEM-01:** `zt_binding_scope_dispose()` frees memory but does not invalidate pointers. Re-use of disposed scopes can cause use-after-free.
- **SEM-02:** `zt_type_param_has_trait()` assumes constraint is `ZT_TYPE_RESULT`. Malformed types bypass trait checking.

### 3.6 HIR Lowering (`compiler/hir/lowering/from_ast.c` — ~800 LOC)

**Responsibilities:** Remove surface ambiguity, normalize named arguments, apply defaults, represent resolved symbols

### 3.7 ZIR Lowering (`compiler/zir/lowering/from_hir.c` — ~1,200 LOC)

**Responsibilities:** Convert HIR to backend-facing ZIR with explicit control flow and structured operands

### 3.8 ZIR Verifier (`compiler/zir/verifier.c` — ~300 LOC)

**Responsibilities:** Validate ZIR invariants before code generation

### 3.9 C Emitter (`compiler/targets/c/emitter.c` — 7,609 LOC)

**Responsibilities:** Translate ZIR to compilable C code

**Critical functions:**
| Line | Function | Risk |
|------|----------|------|
| 48 | `c_canonicalize_type()` | `strncpy` without guaranteed null-termination |
| 128 | `c_buffer_reserve()` | Buffer growth logic |
| 164 | `c_buffer_append_vformat()` | `vsnprintf` double-call pattern |
| 231 | `c_copy_sanitized()` | Symbol name sanitization — collision risk |
| 6047 | `c_emit_check_contract_instruction()` | Where contract code emission |
| 7364 | `c_emit_function_definition()` | Full function emission with cleanup |

**Bugs found:**
- **EMI-01:** `c_buffer_append_vformat()` calls `vsnprintf(NULL, 0)` then `vsnprintf()` real buffer. Not thread-safe; buffer can be reallocated between calls.
- **EMI-02:** `c_canonicalize_type()` uses `strncpy(dest, src, capacity)` without `dest[capacity-1] = '\0'`. If source >= capacity, destination is not null-terminated.
- **EMI-03:** `c_copy_sanitized()` converts both `-` and `_` to `_`. `app.main-test` and `app.main_test` collide to `app_main_test`.
- **EMI-04:** Monomorphization does not deduplicate types with whitespace differences: `list<list<int>>` vs `list<list< int>>` generate separate instances.
- **EMI-05:** FFI shield retain/release may not cover all error paths in extern calls.

### 3.10 Pipeline (`compiler/driver/pipeline.c` — 1,334 LOC)

**Responsibilities:** Orchestrate full compilation, monomorphization limit, native C compilation

**Critical functions:**
| Line | Function | Risk |
|------|----------|------|
| 441 | `zt_compile_project()` | Full pipeline orchestration |
| 549 | stdlib import scan | Textual `strstr("import std.")` — false positives |
| 1112 | `zt_compile_runtime_object()` | Writes to `.ztc-tmp/runtime/` without file lock |
| 1165 | `zt_compile_c_file()` | `compile_cmd[2048]` buffer — truncation with long paths |

**Bugs found:**
- **PIP-01:** `zt_compile_c_file()` builds `compile_cmd[2048]` via `snprintf`. Long project paths + linker flags can truncate, potentially causing command injection via `build.linker_flags`.
- **PIP-02:** stdlib import scan at line 551 uses `strstr(line, "import std.")`. A comment like `-- import std.io` triggers false positive stdlib loading.
- **PIP-03:** `zt_compile_runtime_object()` writes `.ztc-tmp/runtime/zenith_rt.o` without file locking. Parallel builds corrupt the object file.
- **PIP-04:** `zenith.ztproj` is parsed manually; unknown fields are silently ignored.

## 4. Runtime Analysis (`runtime/c/zenith_rt.c` — 5,231 LOC)

### 4.1 Reference Counting (ARC)

**Implementation:** Non-atomic `uint32_t` refcount in heap header

**Critical functions:**
| Line | Function | Risk |
|------|----------|------|
| 865 | `zt_retain()` | `header->rc += 1` — non-atomic, overflow check exists |
| 883 | `zt_release()` | `header->rc -= 1` — non-atomic, underflow check exists |
| 977 | `zt_deep_copy()` | Recursive deep copy — NULL check missing in list case |

**Bugs found:**
- **RUN-01:** `zt_retain()` increments `header->rc` without atomic operations. If FFI creates threads that share `text`/`list` values, race conditions cause incorrect refcounts → double-free or leak.
- **RUN-02:** `zt_deep_copy()` for `ZT_HEAP_LIST_TEXT` (line 1007-1015) does not check if `zt_deep_copy(l->data[i])` returns NULL. NULL is stored in `clone->data[i]` → subsequent access crashes.
- **RUN-03:** `zt_text_from_utf8()` allocates `value` then `value->data`. If second `malloc` fails, `value` leaks (not freed before returning NULL).
- **RUN-04:** `zt_runtime_error_ex()` calls `exit()` without releasing managed values in upper scopes. Memory leaks on panic.

### 4.2 Text Operations

| Line | Function | Risk |
|------|----------|------|
| 533 | `zt_utf8_validate()` | Exists but is NOT called by `zt_text_from_utf8()` |
| 1456 | `zt_text_from_utf8()` | Creates text without UTF-8 validation |
| 1510 | `zt_text_index()` | Byte-level indexing, not grapheme-level |
| 1520 | `zt_text_slice()` | Slice with inclusive end boundary |

**Bugs found:**
- **RUN-09:** `zt_utf8_validate()` exists (line 533, ~200 lines of validation logic) but `zt_text_from_utf8()` (line 1456) does NOT call it. Invalid UTF-8 (overlong sequences, surrogates) can be created as valid `text` values.

### 4.3 Container Operations

| Line | Function | Risk |
|------|----------|------|
| 493 | `zt_normalize_slice_end()` | Inconsistent behavior between text and list slices |
| 1807 | `zt_list_text_get_optional()` | Safe lookup (returns optional) |
| 1794 | `zt_list_i64_get_optional()` | Safe lookup (returns optional) |

**Bugs found:**
- **RUN-07:** `zt_normalize_slice_end()` behaves inconsistently for empty containers. `text_slice` on empty text vs `list_slice` on empty list produce different results for `[0..-1]`.
- **RUN-06:** Template-generated `zt_list_i64_reserve()` uses `new_capacity *= 2`. With > SIZE_MAX/2 elements, this overflows `size_t` → allocation of tiny buffer → heap corruption.

### 4.4 Map Operations

**Implementation:** Open-addressing hash map with linear probing

**Bugs found:**
- **RUN-05:** `zt_map_text_text_new()` allocates `hash_capacity = 8` fixed initial capacity. Maps with >1000 entries degrade to O(n) lookup as load factor increases. No rehashing strategy observed.

### 4.5 Contract Checking

| Line | Function | Risk |
|------|----------|------|
| 1418 | `zt_contract_failed()` | Generic contract failure |
| 1422 | `zt_contract_failed_i64()` | Integer contract failure |
| 1434 | `zt_contract_failed_float()` | Float contract failure |
| 1446 | `zt_contract_failed_bool()` | Bool contract failure |

**Bugs found:**
- **RUN-08:** Contract failure functions use stack buffer of 512 bytes for message formatting. Messages > 512 chars truncate silently.

### 4.6 Host API (File I/O, Process, OS)

| Line | Function | Risk |
|------|----------|------|
| 2991 | `zt_host_default_read_file()` | No path traversal validation |
| 3052 | `zt_host_default_write_file()` | No path traversal validation |
| 3438 | `zt_host_default_process_run()` | Process execution with argument array |

**Bugs found:**
- **STD-03:** `zt_host_default_read_file()` accepts paths like `../../../etc/passwd` without validation. No sandboxing or chroot.
- **STD-02:** `zt_net_connection` does not prevent double-close. Socket fd can leak if `close()` is called twice.

### 4.7 JSON Parser

| Line | Function | Risk |
|------|----------|------|
| 3901 | `zt_json_parse_map_text_text()` | Recursive descent JSON parser |

**Bugs found:**
- **STD-04:** `zt_json_parse_map_text_text()` is recursive. JSON with 10,000+ nesting levels causes C stack overflow.

### 4.8 Shared/Thread-Boundary API

| Line | Function | Purpose |
|------|----------|---------|
| 1141 | `zt_shared_ops` | Vtable for shared handles |
| 1159 | `zt_shared_text_snapshot_value()` | Thread-safe text snapshot |
| 2170 | `zt_thread_boundary_copy_text()` | Deep copy for isolate transfer |

**Design note:** Thread boundary copies use deep copy (not shared references), which is correct for the isolate model. However, the deep copy itself has the NULL check bug (RUN-02).

## 9. Test Coverage Gaps (Zero Coverage)

1. **RC overflow/underflow** — No tests for `retain()` at `UINT32_MAX` or `release()` at `rc=0`
2. **Use-after-free in runtime** — `zt_deep_copy()` NULL deref, `zt_binding_scope_dispose()` pointer reuse
3. **Path traversal / command injection** — Pipeline `compile_cmd` truncation, `read_file` traversal
4. **Monomorphization explosion** — No tests exceeding `build.monomorphization_limit` in practice
5. **FFI with malformed data** — NULL pointers, non-terminated strings from C side
6. **Formatter idempotency** — `format(format(x)) == format(x)` not guaranteed or tested
7. **Parallel builds** — Race conditions in `.ztc-tmp/runtime/zenith_rt.o`
8. **OOM recovery** — No tests forcing `malloc` failure at various allocation points
9. **Invalid UTF-8** — Overlong sequences, surrogates, invalid continuation bytes
10. **Enum payload match E2E** — ZIR lowering for payload enums is partially stubbed
11. **`public var` cross-namespace write** — Multi-file compilation with external write rejection
12. **`dyn Trait` heterogeneous collections** — 1000+ elements with fat pointer dispatch

## 10. Security Attack Surface

### 10.1 Command Injection

- **Vector:** `build.linker_flags` in `zenith.ztproj` injected into `compile_cmd[2048]`
- **Impact:** Arbitrary command execution via `"; rm -rf /"` in linker flags
- **Mitigation needed:** Validate/sanitize linker flags, use `execvp` instead of `system()`

### 10.2 Path Traversal

- **Vector:** `std.fs.read_text("../../../etc/passwd")`
- **Impact:** Read arbitrary files on host filesystem
- **Mitigation needed:** Normalize paths, reject `..` segments, sandbox to project root

### 10.3 Buffer Overflows

- **Vector:** Hex bytes > 512 pairs, string literals > available memory, name paths > 1024 chars
- **Impact:** Stack buffer overflow, memory exhaustion
- **Mitigation needed:** Dynamic buffers, size limits with diagnostics

### 10.4 Denial of Service

- **Vector:** JSON with 10k nesting, 10MB strings, 100k generic instantiations
- **Impact:** Stack overflow, OOM, compilation timeout
- **Mitigation needed:** Recursion limits, size limits, monomorphization enforcement

### 10.5 Race Conditions

- **Vector:** FFI threads sharing managed values, parallel builds
- **Impact:** Double-free, corrupted runtime object, memory corruption
- **Mitigation needed:** Atomic RC for shared values, file locking for build artifacts

## 11. Design Decisions Impacting Tests

| Decision | Impact on Testing |
|----------|-------------------|
| Value semantics | Test `a = b; mutate(b); a == original` invariant |
| No null | Verify parser rejects `null`, only `none` is valid |
| No implicit conversion | Test `const x: int = 1.5` fails; `to_int(1.5)` required |
| Explicit `return` | Functions without `-> Type` return no value |
