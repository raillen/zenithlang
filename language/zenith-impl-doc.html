Zenith — Bytecode VM & Self-Hosting

// Documentação de implementação incremental
C compilador bytecode self-hosting VM

Visão geral

Este documento cobre dois caminhos avançados para o compilador Zenith, em ordem de complexidade crescente. São independentes — você pode fazer só um, os dois, ou fazer o bytecode primeiro e depois o self-hosting.

CaminhoO que produzDificuldadeQuando faz sentido
Bytecode VMSeu próprio formato .zbc + VM em CMédioVocê quer controle total de execução, portabilidade, ou performance melhor que a transpilação para Lua
Self-hostingO compilador Zenith reescrito em ZenithAvançadoA linguagem está madura o suficiente para se expressar; você quer independência do C

Prerequisitos (o que você já precisa ter)

Antes de começar qualquer um dos caminhos, o compilador Zenith em C precisa ter:

  1. Lexer completo — todos os tokens da linguagem reconhecidos
  2. Parser completo — AST gerada corretamente para toda a sintaxe
  3. Binder — resolução de nomes e escopos funcionando
  4. Um backend funcional (a transpilação para Lua serve)
  5. Testes de regressão — você precisa saber quando algo quebra
⚠ Não pule testes
Sem testes de regressão, você não vai saber se o novo backend (bytecode ou self-hosted) produz o mesmo resultado que o original. São seus guardrails.

Parte 1

Bytecode VM Médio

Em vez de transpilar para Lua, o compilador vai emitir um formato binário (.zbc) que uma VM em C interpreta diretamente. Você controla 100% da execução.

Como funciona em linhas gerais

Código Zenith │ ▼ lexer + parser + binder (você já tem) AST │ ▼ novo: emitter bytecode arquivo .zbc ← sequência de bytes com instruções │ ▼ novo: VM em C execução ← loop que lê e executa cada instrução

Passo 1 — Projetar o instruction set

O instruction set é a lista de operações que a sua VM sabe executar. Cada operação é representada por um número de 1 byte chamado opcode. Comece pequeno.

opcodes/opcode.h

typedef enum {
    /* ── Constantes ─────────────────────── */
    OP_CONST,       // empurra constante[idx] na stack
    OP_NIL,         // empurra nil
    OP_TRUE,        // empurra true
    OP_FALSE,       // empurra false

    /* ── Aritmética ─────────────────────── */
    OP_ADD,         // pop b, pop a → push a + b
    OP_SUB,
    OP_MUL,
    OP_DIV,
    OP_NEG,         // negação unária

    /* ── Comparação ─────────────────────── */
    OP_EQ,          // push (a == b)
    OP_LT,          // push (a < b)
    OP_LE,          // push (a <= b)

    /* ── Lógica ─────────────────────────── */
    OP_NOT,
    OP_AND,
    OP_OR,

    /* ── Variáveis ─────────────────────── */
    OP_GET_LOCAL,   // lê variável local [slot]
    OP_SET_LOCAL,   // escreve variável local [slot]
    OP_GET_GLOBAL,  // lê variável global por nome
    OP_SET_GLOBAL,

    /* ── Controle de fluxo ──────────────── */
    OP_JUMP,        // pula N bytes adiante
    OP_JUMP_IF_FALSE, // pula se topo da stack for falsy
    OP_LOOP,        // pula N bytes para trás (for loops)

    /* ── Funções ────────────────────────── */
    OP_CALL,        // chama função com N argumentos
    OP_RETURN,      // retorna valor do topo
    OP_CLOSURE,     // cria closure capturando variáveis

    /* ── Stack ───────────────────────────── */
    OP_POP,         // descarta topo da stack
    OP_PRINT,       // pop e printa (debug)

    OP_HALT,        // encerra execução
} OpCode;
💡 Princípio
Stack-based VM: a maioria das instruções opera no topo da stack — elas fazem pop dos operandos e push do resultado. Isso é mais simples de implementar que uma register-based VM (como Lua usa internamente).

Instruções com operandos

Algumas instruções precisam de dados extras logo após o opcode:

InstruçãoBytes extrasSignificado
OP_CONST1 byte (índice)qual constante do pool
OP_GET_LOCAL1 byte (slot)posição na stack frame
OP_JUMP2 bytes (offset)quantos bytes pular
OP_CALL1 byte (argc)número de argumentos

Passo 2 — Formato do arquivo .zbc

O arquivo bytecode tem uma estrutura simples. Cada chunk representa uma função compilada.

.zbc file layout Header [4 bytes] magic: 0x5A 0x45 0x4E 0x00 ("ZEN\0") [1 byte] version: 0x01 [1 byte] flags: 0x00 Chunk (uma por função/script principal) [4 bytes] name_len ← tamanho do nome em bytes [N bytes] name ← nome da função (UTF-8) [4 bytes] code_len ← número de bytes de código [N bytes] code ← sequência de opcodes + operandos [4 bytes] const_count← número de constantes [N items] constants ← pool de constantes [4 bytes] line_count ← mapeamento para debug [N pairs] lines ← (offset, linha_original) Constant (cada item do pool) [1 byte] type: 0=nil, 1=bool, 2=number, 3=string [N bytes] value (depende do tipo)

Passo 3 — Representar valores

Zenith precisa representar diferentes tipos de valor em tempo de execução. A forma mais eficiente é uma tagged union:

vm/value.h

typedef enum {
    VAL_NIL,
    VAL_BOOL,
    VAL_NUMBER,
    VAL_STRING,
    VAL_FUNCTION,
    VAL_CLOSURE,
} ValueType;

typedef struct {
    ValueType type;
    union {
        bool     boolean;
        double   number;
        ObjString*   string;
        ObjFunction* function;
        ObjClosure*  closure;
    } as;
} Value;

/* Macros de conveniência */
#define IS_NIL(v)     ((v).type == VAL_NIL)
#define IS_BOOL(v)    ((v).type == VAL_BOOL)
#define IS_NUMBER(v)  ((v).type == VAL_NUMBER)
#define AS_BOOL(v)    ((v).as.boolean)
#define AS_NUMBER(v)  ((v).as.number)
#define NIL_VAL       ((Value){VAL_NIL, {.number = 0}})
#define BOOL_VAL(b)   ((Value){VAL_BOOL, {.boolean = b}})
#define NUMBER_VAL(n) ((Value){VAL_NUMBER, {.number = n}})

A stack de execução

A VM usa um array de Value como stack. O topo é controlado por um ponteiro stack_top:

#define STACK_MAX 256

typedef struct {
    Value  stack[STACK_MAX];
    Value* stack_top;      // aponta para slot ACIMA do topo
} VM;

/* Push: escreve o valor e avança o ponteiro */
static inline void push(VM* vm, Value v) {
    *vm->stack_top = v;
    vm->stack_top++;
}

/* Pop: retrocede o ponteiro e retorna o valor */
static inline Value pop(VM* vm) {
    vm->stack_top--;
    return *vm->stack_top;
}

/* Peek: lê sem remover (distance 0 = topo) */
static inline Value peek(VM* vm, int distance) {
    return vm->stack_top[-1 - distance];
}

Passo 4 — O Emitter (AST → bytecode)

O emitter percorre a AST e emite opcodes para o chunk. Ele substitui o backend que transpila para Lua.

emit/emitter.c — exemplo de como compilar expressões

/* Função principal: compila qualquer nó da AST */
static void emit_node(Emitter* e, AstNode* node) {
    switch (node->kind) {

    case NODE_NUMBER_LITERAL:
        // adiciona constante ao pool e emite OP_CONST
        int idx = chunk_add_const(e->chunk,
                     NUMBER_VAL(node->number));
        emit_bytes(e, OP_CONST, (uint8_t)idx);
        break;

    case NODE_BOOL_LITERAL:
        emit_byte(e, node->boolean ? OP_TRUE : OP_FALSE);
        break;

    case NODE_BINARY_EXPR: {
        // compila lado esquerdo, depois direito, depois operação
        emit_node(e, node->binary.left);
        emit_node(e, node->binary.right);
        switch (node->binary.op) {
            case TOK_PLUS:  emit_byte(e, OP_ADD); break;
            case TOK_MINUS: emit_byte(e, OP_SUB); break;
            case TOK_STAR:  emit_byte(e, OP_MUL); break;
            case TOK_SLASH: emit_byte(e, OP_DIV); break;
            case TOK_EQ_EQ: emit_byte(e, OP_EQ);  break;
        }
        break;
    }

    case NODE_VAR_DECL:
        // compila o valor inicial
        emit_node(e, node->var_decl.initializer);
        // registra o slot local (posição na stack frame)
        declare_local(e, node->var_decl.name);
        break;

    case NODE_VAR_EXPR:
        int slot = resolve_local(e, node->name);
        if (slot >= 0) {
            emit_bytes(e, OP_GET_LOCAL, (uint8_t)slot);
        } else {
            // variável global
            emit_load_global(e, node->name);
        }
        break;
    }
}

Compilando if-else (backpatching)

Controle de fluxo exige um truque chamado backpatching: você emite o jump antes de saber para onde ele vai, e depois volta para preencher o offset correto.

case NODE_IF_STMT: {
    // 1. compila a condição
    emit_node(e, node->if_stmt.condition);

    // 2. emite o jump (offset ainda desconhecido → 0xFF 0xFF)
    int then_jump = emit_jump(e, OP_JUMP_IF_FALSE);
    emit_byte(e, OP_POP);  // descarta a condição

    // 3. compila o bloco then
    emit_node(e, node->if_stmt.then_branch);

    int else_jump = emit_jump(e, OP_JUMP);

    // 4. BACKPATCH: agora sabemos onde o then termina
    patch_jump(e, then_jump);
    emit_byte(e, OP_POP);

    // 5. compila o bloco else (se existir)
    if (node->if_stmt.else_branch)
        emit_node(e, node->if_stmt.else_branch);

    patch_jump(e, else_jump);
    break;
}

/* emite jump com placeholder e retorna posição */
static int emit_jump(Emitter* e, uint8_t op) {
    emit_byte(e, op);
    emit_byte(e, 0xFF); // high byte do offset
    emit_byte(e, 0xFF); // low byte
    return e->chunk->code_len - 2;
}

/* preenche o offset real */
static void patch_jump(Emitter* e, int offset) {
    int jump = e->chunk->code_len - offset - 2;
    e->chunk->code[offset]     = (jump >> 8) & 0xFF;
    e->chunk->code[offset + 1] = jump & 0xFF;
}

Passo 5 — A VM e o dispatch loop

O coração da VM é um loop que lê o próximo opcode e executa a instrução correspondente. Isso é chamado de interpreter loop.

vm/vm.c

#define READ_BYTE()  (*ip++)
#define READ_SHORT() (ip += 2, (ip[-2] << 8) | ip[-1])
#define READ_CONST() (chunk->constants[READ_BYTE()])
#define BINARY_OP(op) \
    do { \
        if (!IS_NUMBER(peek(vm, 0)) || !IS_NUMBER(peek(vm, 1))) \
            return runtime_error(vm, "operands must be numbers"); \
        double b = AS_NUMBER(pop(vm)); \
        double a = AS_NUMBER(pop(vm)); \
        push(vm, NUMBER_VAL(a op b)); \
    } while(0)

InterpretResult vm_run(VM* vm, Chunk* chunk) {
    uint8_t* ip = chunk->code;  // instruction pointer

    for (;;) {
        uint8_t instruction = READ_BYTE();
        switch (instruction) {

        case OP_CONST:
            push(vm, READ_CONST());
            break;

        case OP_NIL:   push(vm, NIL_VAL);        break;
        case OP_TRUE:  push(vm, BOOL_VAL(true)); break;
        case OP_FALSE: push(vm, BOOL_VAL(false));break;

        case OP_ADD: BINARY_OP(+); break;
        case OP_SUB: BINARY_OP(-); break;
        case OP_MUL: BINARY_OP(*); break;
        case OP_DIV: BINARY_OP(/); break;

        case OP_EQ: {
            Value b = pop(vm), a = pop(vm);
            push(vm, BOOL_VAL(values_equal(a, b)));
            break;
        }

        case OP_JUMP: {
            uint16_t offset = READ_SHORT();
            ip += offset;
            break;
        }

        case OP_JUMP_IF_FALSE: {
            uint16_t offset = READ_SHORT();
            if (is_falsy(peek(vm, 0))) ip += offset;
            break;
        }

        case OP_GET_LOCAL: {
            uint8_t slot = READ_BYTE();
            push(vm, vm->stack[slot]);
            break;
        }

        case OP_SET_LOCAL: {
            uint8_t slot = READ_BYTE();
            vm->stack[slot] = peek(vm, 0); // não faz pop
            break;
        }

        case OP_POP:   pop(vm);  break;
        case OP_PRINT: print_value(pop(vm)); printf("\n"); break;
        case OP_HALT:  return INTERPRET_OK;
        }
    }
}
ℹ Instruction Pointer
O ip (instruction pointer) é um ponteiro simples para o array de bytecode. Cada READ_BYTE() avança ele em 1 byte. Isso é extremamente rápido.

Passo 6 — Funções e closures

Funções em bytecode funcionam com call frames: cada chamada empilha um frame que guarda o estado local daquela chamada.

vm/vm.h

typedef struct {
    ObjClosure* closure;  // a função sendo executada
    uint8_t*    ip;       // instrução atual NESTE frame
    Value*      slots;    // ponteiro para o início das locais
} CallFrame;

#define FRAMES_MAX 64

typedef struct {
    CallFrame  frames[FRAMES_MAX];
    int        frame_count;
    Value      stack[STACK_MAX];
    Value*     stack_top;
    Table      globals;   // hash table de variáveis globais
    Obj*       objects;   // lista ligada para GC
} VM;

Chamando uma função

static bool call_value(VM* vm, Value callee, int argc) {
    if (!IS_CLOSURE(callee))
        return runtime_error(vm, "can only call functions");

    ObjClosure* closure = AS_CLOSURE(callee);
    if (argc != closure->function->arity)
        return runtime_error(vm, "wrong number of arguments");

    if (vm->frame_count == FRAMES_MAX)
        return runtime_error(vm, "stack overflow");

    CallFrame* frame = &vm->frames[vm->frame_count++];
    frame->closure = closure;
    frame->ip      = closure->function->chunk.code;
    frame->slots   = vm->stack_top - argc - 1;
    return true;
}

Closures — capturando variáveis

Uma closure captura variáveis do escopo externo através de upvalues: referências para valores que podem viver além do frame em que foram declarados.

typedef struct ObjUpvalue {
    Obj        base;
    Value*     location;   // aponta para o valor na stack
    Value      closed;     // valor quando o frame é encerrado
    struct ObjUpvalue* next;
} ObjUpvalue;

typedef struct {
    Obj         base;
    ObjFunction* function;
    ObjUpvalue** upvalues;  // array de upvalues capturados
    int          upvalue_count;
} ObjClosure;

Passo 7 — Garbage Collector básico

A VM precisa liberar memória de strings, closures e outros objetos heap-allocated. O GC mais simples para começar é o mark-and-sweep.

  1. Mark (marcar) A VM percorre todas as raízes (stack, globals, frames) e marca cada objeto alcançável com um bit is_marked = true.
  2. Sweep (varrer) A VM percorre a lista ligada de todos os objetos alocados. Objetos não marcados são liberados. Objetos marcados têm o bit resetado para a próxima coleta. 
void gc_collect(VM* vm) {
    /* 1. Mark — começa pelas raízes */
    for (Value* slot = vm->stack; slot < vm->stack_top; slot++)
        gc_mark_value(vm, *slot);

    for (int i = 0; i < vm->frame_count; i++)
        gc_mark_object(vm, (Obj*)vm->frames[i].closure);

    gc_mark_table(vm, &vm->globals);

    /* 2. Sweep — libera o que não foi marcado */
    Obj** obj = &vm->objects;
    while (*obj != NULL) {
        if (!(*obj)->is_marked) {
            Obj* unreached = *obj;
            *obj = unreached->next;
            obj_free(unreached);
        } else {
            (*obj)->is_marked = false;
            obj = &(*obj)->next;
        }
    }
}
💡 Quando chamar o GC?
Não chame a cada alocação — é muito caro. Use um threshold: quando o total de bytes alocados ultrapassar um limite (ex: 1MB inicial), trigger um ciclo de coleta.

Parte 2

Self-Hosting Avançado

Self-hosting significa que o compilador Zenith é escrito em Zenith. Em vez de um compilador em C que compila Zenith, você vai ter um compilador em Zenith que compila Zenith — e ele pode compilar a si mesmo.

Por que isso é especial?

Um compilador self-hosted é uma prova de maturidade da linguagem. Se Zenith consegue expressar seu próprio compilador — com todas as estruturas de dados, algoritmos e IO necessários — ela está completa o suficiente para uso geral.

⚠ Prerequisito crítico
Self-hosting cria um problema de bootstrap circular: você precisa de um compilador Zenith para compilar o compilador Zenith. A solução é o processo de 3 estágios descrito abaixo.

Passo 1 — O que a linguagem precisa ter

Antes de escrever o compilador em Zenith, ela precisa suportar:

FuncionalidadePara que serve no compiladorUrgente?

Arrays dinâmicosBuffer de tokens, lista de nós filhos✅ Sim
Strings com indexaçãoLexer precisa ler caractere por caractere✅ Sim

IO de arquivoLer o arquivo fonte, escrever o output✅ Sim

Passo 2 — O bootstrap em 3 estágios

O bootstrap é o processo de criar uma cadeia de compiladores até o compilador final ser capaz de compilar a si mesmo. Chamamos os compiladores de stage 0, stage 1 e stage 2.

Stage 0 — O compilador Zenith original em C (você já tem este) │ │ compila ▼ zenith_compiler.zen ← o compilador reescrito em Zenith │ │ Stage 0 compila zenith_compiler.zen ▼ Stage 1 — binário do compilador, gerado pelo Stage 0 │ │ Stage 1 compila zenith_compiler.zen novamente ▼ Stage 2 — binário do compilador, gerado pelo Stage 1 Validação: Stage 1 e Stage 2 devem ser bit-a-bit idênticos. Se forem, o compilador está correto. 🎉
💡 Por que comparar Stage 1 e Stage 2?
Se um compilador compilar a si mesmo e produzir um output idêntico ao da compilação anterior, você tem uma garantia forte de correção. Essa técnica se chama Diverse Double-Compiling.

Comandos do bootstrap

# Stage 0: compilador em C compila o compilador Zenith
$ zenithc --output stage1 src/compiler/zenith_compiler.zen

# Stage 1: compilador Zenith compila a si mesmo
$ ./stage1 --output stage2 src/compiler/zenith_compiler.zen

# Validação: os dois binários devem ser iguais
$ diff stage1 stage2
# (nenhuma saída = sucesso!)

Passo 4 — O Lexer em Zenith

O lexer lê o arquivo fonte caractere por caractere e produz uma lista de tokens. Em Zenith, vai parecer algo assim:

// Definição de tipos que o lexer precisa
struct Token {
    kind:    TokenKind,
    lexeme:  String,
    line:    Int,
}

struct Lexer {
    source:  String,
    start:   Int,   // início do token atual
    current: Int,   // posição atual de leitura
    line:    Int,
}

fn lexer_new(source: String) -> Lexer {
    return Lexer { source, start: 0, current: 0, line: 1 }
}

fn next_token(l: *Lexer) -> Token {
    skip_whitespace(l)
    l.start = l.current

    if is_at_end(l) { return make_token(l, TOKEN_EOF) }

    let c = advance(l)  // lê e avança

    if is_alpha(c) { return scan_identifier(l) }
    if is_digit(c) { return scan_number(l) }

    match c {
        '(' => return make_token(l, TOKEN_LPAREN),
        ')' => return make_token(l, TOKEN_RPAREN),
        '+' => return make_token(l, TOKEN_PLUS),
        '=' => {
            if match_char(l, '=') { return make_token(l, TOKEN_EQ_EQ) }
            return make_token(l, TOKEN_EQ)
        },
        // ... demais casos
    }
}

Passo 5 — O Parser em Zenith

O parser usa o lexer para produzir a AST. A técnica mais simples — e que você provavelmente já usa no compilador C — é o parser recursivo-descendente.

struct Parser {
    lexer:    Lexer,
    current:  Token,
    previous: Token,
    had_error: Bool,
}

// Cada regra gramatical vira uma função
fn parse_expr(p: *Parser) -> *AstNode {
    return parse_assignment(p)
}

fn parse_assignment(p: *Parser) -> *AstNode {
    let left = parse_equality(p)
    if match(p, TOKEN_EQ) {
        let right = parse_assignment(p)  // recursão (right-associative)
        return node_assign(left, right)
    }
    return left
}

fn parse_equality(p: *Parser) -> *AstNode {
    let left = parse_comparison(p)
    while match_any(p, [TOKEN_EQ_EQ, TOKEN_BANG_EQ]) {
        let op = p.previous
        let right = parse_comparison(p)
        left = node_binary(op, left, right)
    }
    return left
}

// Padrão: o mesmo cascadeamento de precedência que o parser C
// parse_expr → parse_assignment → parse_equality
// → parse_comparison → parse_addition → parse_multiplication
// → parse_unary → parse_primary
ℹ A AST em Zenith
Você vai precisar de uma tagged union para nós da AST — da mesma forma que o compilador C tem. Em linguagens com structs mas sem enums-with-data, use um campo kind: Int + uma union ou um campo data: Any.

Passo 6 — O Emitter em Zenith

O emitter em Zenith vai emitir o mesmo bytecode que o emitter em C emite. A lógica é idêntica — a diferença é só que agora ela está escrita em Zenith.

struct Emitter {
    chunk:    Chunk,
    locals:   Array[Local],
    scope_depth: Int,
}

fn emit_node(e: *Emitter, node: *AstNode) {
    match node.kind {
        NODE_NUMBER =>
            emit_const(e, value_number(node.number)),
        NODE_BINARY =>
            emit_binary(e, node),
        NODE_IF =>
            emit_if(e, node),
        NODE_WHILE =>
            emit_while(e, node),
        NODE_CALL =>
            emit_call(e, node),
        // ... demais casos espelham o emitter em C
    }
}

fn emit_while(e: *Emitter, node: *AstNode) {
    let loop_start = e.chunk.code_len  // guarda onde o loop começa
    emit_node(e, node.while_stmt.condition)
    let exit_jump = emit_jump(e, OP_JUMP_IF_FALSE)
    emit_byte(e, OP_POP)
    emit_node(e, node.while_stmt.body)
    emit_loop(e, loop_start)   // volta para o início
    patch_jump(e, exit_jump)
    emit_byte(e, OP_POP)
}

Passo 7 — Validação e o momento do bootstrap

Quando os três componentes (lexer, parser, emitter) estiverem funcionando em Zenith, chegou a hora do bootstrap:

  1. Compile o compilador Zenith com Stage 0 (C) 
    $ zenithc -o stage1 src/compiler/main.zen
  2. Use Stage 1 para compilar o compilador novamente 
    $ ./stage1 -o stage2 src/compiler/main.zen
  3. Compare os binários 
    $ sha256sum stage1 stage2
# Se os hashes forem iguais: ✅ self-hosting concluído
  4. A partir de agora: Stage 0 pode ser aposentado O compilador Zenith é oficialmente self-hosted. Novos features são implementados em Zenith e compilados com a versão anterior.
🔴 Se Stage 1 ≠ Stage 2
Há um bug no emitter do compilador Zenith. O compilador produz outputs diferentes dependendo de como foi compilado — geralmente causado por comportamento undefined (uninitialised memory, overflow de int) no compilador em C, ou por uma inconsistência no próprio emitter Zenith.
